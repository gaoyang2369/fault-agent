# faultAgent 项目快速上手

## 1. 项目要解决什么问题

faultAgent 是一个面向工业驱动系统历史数据的只读诊断后端。当前首个闭环资产是
`G120-1`，它在领域中表示一套 `DriveSystem`（驱动系统），由独立的 `Inverter`
（变频器）和 `Motor`（电机）组成，不代表已经确认的电机型号。

系统的目标链路是：接收资产和时间范围查询，读取 `real_data` 历史数据，检查数据
质量，识别设备上报事件和确定性异常，关联证据与知识，形成故障假设和检查建议，
最终由已认证的工程师确认故障。设备故障码只是 `ReportedEvent`，不能直接成为
`ConfirmedFault`。

当前项目处在工程底座和核心契约阶段，还不是完整可用的诊断产品。代码没有设备控制
能力，也没有让 LLM 决定权限、工业阈值、诊断置信度或确认故障。

## 3. 架构怎么理解

项目采用“模块化单体 + 独立 Agent worker”。业务能力在一个代码仓库中按领域模块隔离，
HTTP API 和 Agent tools 将来必须复用同一套 application service；Agent tool 不允许绕过
服务层访问数据库。

```text
HTTP API ─┐
          ├─> application service ─> domain model / policy ─> infrastructure port
Agent tool┘                                                └─> real_data（只读）

Agent worker：独立进程，只负责计划和编排，不直接访问数据库
```

核心目录职责如下：

```text
apps/
  api/                 FastAPI 装配、HTTP 路由
  agent_worker/        独立 Agent 进程入口（当前为占位）
modules/
  asset/               DriveSystem、Component、信号定义和资产到源表的映射
  telemetry/           查询命令、结果、数据质量、只读 real_data 适配
  evidence/            不可变 Evidence 快照和必须绑定证据的 Claim
  diagnosis/           ReportedEvent、Anomaly、Hypothesis、ConfirmedFault 等模型
  iam/                  权限模块（待实现）
  knowledge/            知识模块（待实现）
  report/               报告模块（待实现）
  audit/                审计模块（待实现）
shared/                 跨模块的时间、身份标识、请求上下文和错误契约
contracts/              对外 JSON Schema、示例及生成脚本
tools/                  运维/研发工具，目前包含只读数据摸底 CLI
tests/                  unit、integration、contract 三层测试
doc/m0/                 M0 范围、领域规则、数据事实和任务拆分
```

模块内部优先遵循以下依赖方向：

```text
domain <- application <- infrastructure / apps
```

- `domain` 只表达业务概念和不变量，不出现源表名称、SQL 或 HTTP 细节。
- `application` 编排资产解析、授权和查询端口，是 HTTP 与 Agent 复用的入口。
- `infrastructure` 实现数据库、外部系统或临时兼容适配。
- `apps` 只负责进程和协议装配，不承载领域判断。

## 4. telemetry 迁移后的分层

- `modules/telemetry/domain/` 与 `application/` 定义唯一公开契约，调用者只提交
  `asset_id` 或 `asset_code`。
- `modules/telemetry/infrastructure/real_data_repository.py` 原生实现应用查询端口，内部才使用
  `device_name` / `inverter_name`，并保留固定只读 SQL、信号白名单、显式源时区、超时、扫描和
  返回上限等约束。
- Task 3.1 的旧公开模型、旧服务和 `LegacyRealDataBackend` 已删除，不再需要新旧模型转换。

## 5. 一次公开查询将怎样流动

当前已实现的应用层调用顺序是：

1. 构造 `TelemetryQueryCommand`，且只能提供 `asset_id`、`asset_code` 二选一。
2. 由可信边界注入 `RequestContext`；不能接受 LLM/tool payload 自报的用户或角色。
3. `AssetSourceResolver` 解析公开资产，并在内部取得 `RealDataSourceLocator`。
4. `TelemetryAuthorizationPolicy` 对命令、资产和可信上下文做确定性授权或改写。
5. `TelemetryQueryBackend` 执行固定、参数化、只读查询并归一化时间。
6. 返回不包含 `device_name`、`inverter_name` 等私有定位字段的公开结果。

当前第 4 步只为 HTTP 游客提供最低安全边界，尚未实现真实认证和工程师/管理员策略，因此
这条链路仍不应被视为生产权限闭环。

## 6. 本地运行与验证

```powershell
python -m pip install -e ".[dev]"
ruff format --check .
ruff check .
mypy
pytest
```

启动 API：

```powershell
uvicorn apps.api.main:app --reload
```

检查存活状态：

```powershell
Invoke-RestMethod http://127.0.0.1:8000/health
```

运行只读数据摸底需要显式配置环境变量，尤其是 DSN、查询超时和限制。不要提交真实
`.env` 或生产凭据。示例配置见 `.env.example`，入口为：

```powershell
python -m tools.profile_real_data
```

## 8. 接下来应该做什么

建议按依赖顺序推进，而不是先接入 LLM：

1. Task 4 领域契约和 Task 5 IAM 骨架已经完成；继续实现知识与故障码结构化接口。知识文档只能

   文档版本和来源位置。
2. 实现规则框架和数据质量门控。未确认单位时关闭单位相关规则，不添加温度、负载、电流、
   转速工业阈值。
3. 建立 Evidence/Claim 持久化、诊断服务和全链路审计；每个 DiagnosisRun 保存规则、知识
   和模型版本。
4. 最后实现单 Agent 编排、答案生成与报告；LLM 只做受约束的理解和表达，不做权限、规则、
   置信度或 ConfirmedFault 决策。

## 9. 开发时必须持续检查的边界

- `real_data` 永远只读；SQL 参数化，信号列来自显式白名单，并配置超时与结果上限。
- 源时区来自配置，不能猜测；未知单位保持 `null`，不能静默补单位。
- Observation、ReportedEvent、Anomaly、Hypothesis、ConfirmedFault 是不同概念。
- 任何实质性 Claim 都要引用 Evidence，并保存源记录 id、时间范围和版本。
- ConfirmedFault v1 只能由已认证的人确认。
- 公共 HTTP API 与 Agent tools 必须复用 application service，并在每次调用确定性授权。
- 所有行为变更同步更新契约与测试；未决领域/安全问题分别标记 `TODO-DOMAIN` 和
  `TODO-SECURITY`。
