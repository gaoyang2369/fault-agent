> 更新日期：2026-07-19

## 1. 先用一句话理解项目

faultAgent 是一个面向工业驱动系统历史数据的只读诊断后端。它把资产、遥测、数据质量、设备
上报事件、证据、故障假设和人工确认分开建模，目标是形成“结论有证据、过程可审计、权限不可由
LLM 绕过”的诊断链路。

当前资产编码 `G120-1` 表示一套 `DriveSystem`，其中 `Inverter`（变频器）和 `Motor`（电机）是
两个独立部件。它不是已确认的电机型号。`fault_code` 或 `alarm_code` 只会形成
`ReportedEvent`，也不能直接成为根因或 `ConfirmedFault`。

当前代码已经能完成受权的历史遥测查询，但还不能完成端到端故障诊断、知识检索或报告生成，也
没有任何设备控制能力。

## 3. 架构地图

项目采用“模块化单体 + 独立 Agent worker”，不是微服务。HTTP API 和 Agent Tool 必须复用
application service；Tool 不得直接访问数据库。

```text
HTTP API ─┐
          ├──> application service ──> domain / policy ──> infrastructure port
Agent Tool┘                                                   └──> real_data（只读）

Agent worker：独立进程，只负责编排；当前尚未实现 Orchestrator 和 LLM Adapter。
```

主要目录：

```text
apps/
  api/                     FastAPI 工厂、组合根和 HTTP 路由
  agent_worker/            独立 worker 入口及复用应用服务的遥测 Tool
modules/
  iam/                     认证端口、可信上下文和确定性授权策略
  asset/                   DriveSystem、Component、SignalDefinition 和源定位映射
  telemetry/               查询命令、结果、质量门控、应用服务和 real_data 适配器
  evidence/                Evidence 快照与 Claim 证据约束
  diagnosis/               事件、异常、假设、建议、人工确认故障和诊断结果模型
  knowledge/ report/ audit/尚未实现的后续模块边界
shared/                     公共标识、请求上下文、时间和错误契约
contracts/                  生成的 JSON Schema、契约示例与生成脚本
tools/                      real_data 只读摸底 CLI
tests/                      unit、integration、contract 三层测试
doc/m0/                     M0 范围、领域决策、数据事实和 Task 定义
```

模块内依赖方向是 `domain <- application <- infrastructure/apps`。领域层不应出现 HTTP、SQL、
源表字段或数据库连接；基础设施层也不应决定用户权限。

本次已删除仅含 `.gitkeep` 的预建目录。以后实现知识、规则、审计或部署能力时，再根据实际代码
创建所需目录，不提前制造空的架构层级。

## 4. 一次遥测请求怎样流动

1. HTTP 或 Tool 构造严格的 `TelemetryQueryCommand`，公开身份只能是 `asset_id` 或
   `asset_code` 二选一，不能传 SQL、源设备名、用户、角色或资产范围。
2. HTTP 认证后端或 Agent worker 的受信会话边界生成 `TrustedPrincipal`，再创建
   `RequestContext`。
3. `AssetSourceResolver` 解析稳定资产身份；授权策略先依据公开资产和可信上下文做确定性判断。
4. 授权通过后才读取私有的 `RealDataSourceLocator`，取得 `device_name/inverter_name` 映射。
5. `RealDataRepository` 使用内置 SELECT、参数和信号白名单查询，只把显式配置的源时区归一为
   `observed_at`，再执行范围复筛、去重、聚合和数据质量汇总。
6. 返回 `TelemetryQueryResult`；公开结果不暴露源定位或数据库信息。

游客只能查询白名单资产、授权时钟最近一小时、60 秒或更粗的聚合数据，并且不能查询受限状态字。
工程师只能访问显式分配的资产；管理员可访问全部本地资产。未装配策略时默认拒绝。

## 5. 五分钟本地上手

要求 Python 3.12+。在仓库根目录执行：

```powershell
python -m pip install -e ".[dev]"
ruff format --check .
ruff check .
mypy
pytest
```

启动 API（健康检查不需要数据库）：

```powershell
uvicorn apps.api.main:app --reload
Invoke-RestMethod http://127.0.0.1:8000/health
```

`POST /v1/telemetry/queries` 会按需创建真实遥测仓储，因此必须设置 `.env.example` 列出的数据库、
源时区、扫描上限、返回上限和数据质量配置。应用不会自动读取 `.env`；请通过进程环境或部署系统
注入。不要提交真实凭据。

数据摸底工具的入口是：

```powershell
python -m tools.profile_real_data
```

该工具只接受固定表 `real_data` 和内部 SELECT，不接受调用者传入 SQL。默认输出
`real_data_profile.json` 与 `real_data_profile.md`。重新生成报告前应确认目标确实是允许访问的只读
数据源。

## 8. 对既有 Task 的检查结论

本次对照 `doc/m0/11_codex_tasks.md`、领域契约、IAM 说明、实现和测试进行了检查。

- 未发现 Critical 或 High 级问题。API 与 Agent Tool 都复用 `TelemetryQueryService`，Agent Tool
  没有直接访问数据库；公开命令拒绝身份、角色、源定位和原始 SQL；源表访问保持只读。
- Medium：当前所谓 IAM 是可替换骨架，不是生产认证。Bearer 后端是内存 Fixture，资产分配来自
  进程配置，尚无令牌签名/撤销、分配变更审计和 PolicyDecision 记录。
- Medium：Task 3 的数据库集成测试使用内存执行器，没有在自动测试中验证真实 MySQL 的只读会话、
  超时和方言行为；上线前需要受控的非生产数据库测试。
- Medium：`create_time` 只用于带配置缓冲的 SQL 粗筛，其时区、含义以及与 `observed_at` 的最大偏差
  尚未由现场数据契约确认。当前代码会在 Python 中精确复筛并报告扫描截断，这属于安全降级。
- Low：Task 4 已冻结第一版模型和 JSON Schema，但诊断状态机、持久化和版本化写入流程尚未实现；
  不能把“模型存在”理解成“诊断服务已完成”。
- Low：数据摸底结果只是 2026-07-16 快照。未知单位保持 `null`、源时区不猜测、异常数值不解释为
  工业故障，这些限制需要在后续实现中继续保持。

## 9. 开发红线

- `real_data` 永远只读；只使用固定、参数化 SQL、信号白名单、查询超时和结果上限。
- 源时区必须来自配置；未知单位保持 `null`，不能静默赋值或据此启用规则。
- Observation、ReportedEvent、Anomaly、Hypothesis、ConfirmedFault 必须保持独立。
- LLM 不得决定权限、工业阈值、诊断置信度或创建 ConfirmedFault。
- v1 的 ConfirmedFault 必须由已认证人员确认；所有实质性 Claim 必须引用 Evidence。
- 每次行为变更都要同步更新契约与测试；未决领域和安全问题分别标记 `TODO-DOMAIN`、
  `TODO-SECURITY`。
